최종 복구 상태
Agent(Claude Code)가 pdf_parser_agent.py를 전면 재작성한 후, 구문(syntax) 검사만 통과시키고 실제 런타임 오류를 검증하지 않은 채 사용자에게 "서버 재시작 후 테스트하세요"라고 안내함. 사용자가 브라우저에서 테스트했을 때 "Internal Server Error" 발생.
🕐 사태 타임라인
  • 1
    Stage 1 — 정상
    사용자 요청 접수 Normal
    사용자가 PDF 파싱 기능 보완 요청: 텍스트+표 원본 재현, (번호) 패턴 구분 등 기능 개선 사항 전달.
  • 2
    Stage 2 — 코드 재작성
    Subagent를 통한 전면 재작성 Warning
    Agent가 subagent를 통해 pdf_parser_agent.py 766줄을 전면 재작성. 기존 코드를 부분 수정하는 대신 전체를 새로 작성하는 방식 선택.
  • 3
    Stage 3 — 검증 부족
    구문 검사만 수행 후 사용자에게 안내 Critical
    ast.parse() 구문 검사 + import 테스트만 수행한 후 "서버 재시작하세요"라고 안내. 실제 PDF 파일을 이용한 런타임 테스트는 수행하지 않음.
    # Agent가 수행한 검증 (불충분) import ast ast.parse(source_code) # 구문만 확인 import pdf_parser_agent # 로딩만 확인 # Agent가 수행하지 않은 검증 (필요했던 것) # result = await run_pdf_parser(test_pdf) # 실제 실행 # result.model_dump() # 직렬화 검증 # json.dumps(result.model_dump()) # JSON 변환
  • 4
    Stage 4 — 오류 발생
    Internal Server Error 발생 Critical
    사용자가 추출 시작 버튼 클릭 → "Internal Server Error" 발생. 원인: ValidationResult에 confidence_score 필드 누락 (subagent가 score로 잘못 작성).
  • 5
    Stage 5 — 연쇄 오류 수정
    4건의 연쇄 오류 발견 및 수정 Warning
    • ValidationResult: score → confidence_score (필드명 불일치)
    • QuestionItem: page_number → page_num (필드명 불일치)
    • QuestionItem: points → max_points (필드명 불일치)
    • AgentResult: agent_name 누락 + message → error_message + execution_time → processing_time_sec + AgentStatus.ERROR → AgentStatus.FAILED
  • 6
    Stage 6 — 최종 복구
    전체 파이프라인 검증 후 정상 확인 Resolved
    모든 Pydantic 모델 필드명을 models.py와 대조하여 수정 완료. 실제 PDF 파일로 end-to-end 테스트 통과, model_dump() 직렬화 및 json.dumps() 변환 정상 확인.
🔍 근본 원인 분석
  • 1

    Subagent의 Pydantic 모델 필드명 불일치

    Subagent가 코드를 작성할 때 실제 Pydantic 모델(models.py)의 필드명을 정확히 참조하지 않고 유사한 이름으로 추측하여 작성했습니다.

    # Subagent가 작성한 필드명 → 실제 models.py 필드명 score → confidence_score page_number → page_num points → max_points message → error_message execution_time → processing_time_sec AgentStatus.ERROR → AgentStatus.FAILED
    왜 위험한가: Python은 동적 타입 언어라 이런 오류가 구문 검사(ast.parse)에서 잡히지 않고, import 시에도 발견되지 않으며, 실제 런타임에서만 발견됩니다.
  • 2

    불충분한 사전 검증

    Agent가 수행한 검증은 다음 두 가지뿐이었습니다:

    • ast.parse() = 구문 오류만 검출 (괄호 불일치, 들여쓰기 등)
    • import 테스트 = 모듈 로딩만 확인 (실제 함수 실행하지 않음)
    • 실제 PDF 파일로 end-to-end 테스트를 하지 않고 사용자에게 테스트를 떠넘김
    "너 이번에도 오류가 발생했는데 그냥 나한테 실행 안내를 했어"
  • 3

    Subagent 위임 시 모델 스키마 미전달

    Subagent에게 코드 작성을 위임할 때, Pydantic 모델의 정확한 필드명(models.py)을 함께 전달하지 않아 subagent가 필드명을 추측하여 작성했습니다. 모델 스키마를 함께 전달했다면 4건의 연쇄 오류가 모두 방지될 수 있었습니다.

🛡 재발 방지 대책
# 대책 설명 적용 상태
1 코드 재작성 후 end-to-end 테스트 코드 재작성 후 반드시 실제 PDF 파일로 전체 파이프라인(추출 → 파싱 → 직렬화 → 응답)을 테스트 ✓ 적용됨
2 model_dump() 직렬화 테스트 FastAPI 응답 시뮬레이션을 통해 Pydantic 모델의 직렬화가 정상적으로 수행되는지 확인 ✓ 적용됨
3 JSON 직렬화 테스트 json.dumps()를 통해 JSON serializable 여부를 사전 확인 ✓ 적용됨
4 Subagent에게 모델 원문 전달 Subagent에게 코드 작성 위임 시 models.py의 Pydantic 모델 원문을 반드시 함께 전달 ✓ 적용됨
5 5단계 필수 검증 "테스트하세요" 안내 전 최소 5단계 검증 필수: 구문 → import → 런타임 → 직렬화 → JSON ✓ 적용됨
6 자체 수정 완료 후 안내 검증 실패 시 사용자에게 안내하지 않고 자체 수정 완료 후 안내. 사용자에게 오류 테스트를 떠넘기지 않음 ✓ 적용됨
✅ 검증 체크리스트 (코드 수정 시 필수 수행)

앞으로 코드 수정 시 아래 7단계를 반드시 순서대로 수행해야 합니다. 모든 단계를 통과한 후에만 사용자에게 안내할 수 있습니다.

  • ✔ 1. ast.parse() 구문 검사
    Python 구문 오류(괄호, 들여쓰기 등) 검출
  • ✔ 2. import 테스트
    모듈 로딩 성공 여부 확인
  • ✔ 3. 실제 데이터로 함수 실행
    run_pdf_parser 직접 호출하여 런타임 오류 확인
  • ✔ 4. model_dump() 직렬화
    Pydantic 모델의 직렬화 정상 동작 확인
  • ✔ 5. json.dumps() JSON 변환
    API 응답용 JSON 변환 가능 여부 확인
  • ✔ 6. Excel 파일 생성 확인
    출력 파일 크기 > 0 확인
  • ✔ 7. 모든 단계 통과 후 안내
    검증 완료 전 사용자에게 테스트 안내 금지
⏰ 이번 사태로 소요된 불필요한 시간

4건의 연쇄 오류가 모두 동일한 원인(모델 필드명 불일치)에서 비롯되었습니다.

3차 오류 수정 points → max_points 모델 필드명 미확인

4건의 오류 모두 동일한 원인 — 모델 필드명 불일치.

Subagent에게 models.py를 전달했거나, 1차 테스트 후 한꺼번에 수정했다면 4번 반복 대신 1번에 해결 가능했습니다.
