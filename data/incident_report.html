최종 복구 상태
CDP AI Platform 개발 과정에서 프론트엔드 진입점 파일인 frontend/index.html이 총 3차례 손상되었습니다. 각 손상은 서로 다른 원인(JS 문법 오류 삽입, replace 버그로 인한 파일 팽창, 인코딩 미지정으로 인한 파일 소실)에 의해 발생하였으며, 캐시 복구 시도 역시 CP949 인코딩 문제로 한국어 콘텐츠가 소실되는 부작용을 낳았습니다. 최종적으로 HTML 파일을 전면 재건(rebuild)하는 방식으로 복구가 완료되었습니다.
🕐 사태 타임라인
  • 1
    Stage 1 — 정상
    정상 상태 Normal
    frontend/index.html 파일이 정상적으로 존재하며, CDP AI Platform의 모든 기능이 브라우저에서 정상 로딩됨. 파일 크기 약 44KB, UTF-8 인코딩.
  • 2
    Stage 2 — 1차 손상
    JS 문법 오류 삽입 Warning
    파일 수정 과정에서 두 가지 JavaScript 문법 오류가 삽입됨:
    • 중복 키워드: async async function 형태의 잘못된 함수 선언
    • 불완전한 정규식: /\/g (닫는 슬래시 누락으로 파싱 오류)

    브라우저 콘솔에 SyntaxError 발생, 화면 전체 블랭크.
  • 3
    Stage 3 — 2차 손상
    파일 팽창 (44KB → 40MB) Critical
    오류 수정 시도 중 Python 스크립트의 str.replace() 호출에서 치명적인 버그 발생:
    # 버그 코드 — 빈 문자열을 찾아 매 위치마다 new_block 삽입 content.replace('', new_block) # Python에서 빈 문자열('')은 모든 문자 사이에 매칭됨 # 44KB 파일의 수만 개 위치 x new_block 크기 = 40MB 팽창
    44KB였던 파일이 약 40MB로 팽창. 브라우저 로딩 불가 및 에디터 응답 불가 상태 발생.
  • 4
    Stage 4 — 3차 손상
    UnicodeEncodeError → 0 bytes 파일 Critical
    팽창된 파일을 수동으로 정리 후 재저장 시, encoding 파라미터 미지정으로 인해:
    # 문제 코드 — Windows 환경에서 기본 인코딩은 CP949 with open('frontend/index.html', 'w') as f: f.write(content) # 한국어 문자 → UnicodeEncodeError # open()이 파일을 먼저 생성(truncate)한 후 write()에서 오류 발생 # 결과: 0 bytes 빈 파일만 남음
    UnicodeEncodeError: 'cp949' codec can't encode character 발생. 파일이 이미 truncate된 상태에서 예외 발생 → 0 bytes 빈 파일만 남음.
  • 5
    Stage 5 — 캐시 복구 시도
    CP949 인코딩 캐시 복구 — 한국어 소실 Partial
    브라우저 캐시 및 에디터 히스토리에서 이전 버전을 복구하였으나, 해당 캐시가 CP949 인코딩으로 저장되어 있어 한국어 텍스트가 모두 깨지거나 소실됨. UI 레이아웃은 복원되었으나 메뉴 라벨, 버튼 텍스트 등 모든 한국어 콘텐츠 누락. 불완전한 복구 상태로는 서비스 재개 불가 판단.
  • 6
    Stage 6 — 최종 복구
    HTML 전면 재건으로 복구 완료 Resolved
    원본 설계 문서와 기능 명세를 기반으로 frontend/index.html을 처음부터 재건(rebuild).
    # 올바른 코드 — encoding 명시적 지정 with open('frontend/index.html', 'w', encoding='utf-8') as f: f.write(content)
    모든 기능, 한국어 콘텐츠, UI 레이아웃 정상 복구. 이후 index.html.backup 상시 유지 정책 도입.
🔍 근본 원인 분석
  • 1

    파일 쓰기 시 encoding='utf-8' 미지정

    Windows 환경에서 Python의 open() 함수는 기본 인코딩으로 시스템 로케일(CP949/EUC-KR)을 사용합니다. 한국어 문자가 포함된 파일을 CP949로 인코딩 시 지원되지 않는 유니코드 문자가 존재하면 UnicodeEncodeError가 발생합니다. 더 치명적인 문제는 open(path, 'w') 호출 시점에 파일이 이미 truncate(빈 파일로 초기화)되므로, 이후 write()에서 예외가 발생해도 원본 파일 내용은 이미 소실된다는 점입니다.

  • 2

    비원자적(Non-atomic) 파일 쓰기

    원자적 쓰기란 임시 파일에 먼저 내용을 기록한 후, 성공 시에만 원본 파일에 rename하는 방식입니다. 이 패턴을 사용하지 않으면, 쓰기 도중 오류 발생 시 원본 파일이 파괴되거나 불완전한 상태가 됩니다. Stage 3~4의 손상은 모두 비원자적 쓰기로 인해 원본 파일이 복구 불가능하게 손상된 사례입니다.

  • 3

    수정 전 기존 파일 내용 미확인 및 replace() 대상 미검증

    str.replace(old, new) 호출 전에 old 문자열이 실제로 존재하는지 확인하지 않았습니다. 특히 빈 문자열('')은 Python에서 모든 문자 경계에 매칭되므로, content.replace('', new_block)은 파일 전체에 걸쳐 수만 번 삽입 연산을 수행하게 됩니다. 또한 수정 전 현재 파일 상태(인코딩, 내용 정합성)를 확인하는 단계가 누락되어 있었습니다.

# 대책 설명 적용 상태
1 encoding='utf-8' 항상 명시 모든 파일 I/O에서 open(path, 'w', encoding='utf-8') 형태로 인코딩을 명시적으로 지정. Windows/Linux 환경 차이 무관하게 동작 보장. ✓ 적용됨

3 replace() 전 대상 문자열 존재 확인 if target_str in content: 조건 확인 후 replace 실행. 빈 문자열이나 None을 replace 대상으로 사용하는 코드 금지. ✓ 적용됨
4 수정 후 JS 문법 검증 파일 저장 후 async async, function function 등 중복 키워드 및 백틱 홀수 여부를 정규식으로 자동 검사. 검증 실패 시 저장 중단. ✓ 적용됨
5 index.html.backup 상시 유지 배포 전 및 수정 직전 index.html.backup 파일을 자동 생성. restore_frontend.bat 스크립트로 1초 내 복구 가능. ✓ 적용됨
6 검증 통과 후에만 사용자 안내 파일 수정 완료 메시지 출력 전에 반드시 파일 크기, 인코딩, JS 문법 검증 3단계를 통과해야 함. 실패 시 오류 메시지와 롤백 안내 출력. ✓ 적용됨
🚨 긴급 복구 방법
⚠️ index.html이 손상된 경우 아래 방법 중 하나를 사용하여 즉시 복구하세요.

  1. restore_frontend.bat 실행 (권장)

    프로젝트 루트의 restore_frontend.bat를 더블클릭하거나 CMD에서 실행합니다. 백업 파일(index.html.backup)에서 자동 복원하며 인코딩 검증까지 수행합니다.

    REM 프로젝트 루트에서 실행 restore_frontend.bat

  2. 수동 복구: index.html.backup 복사

    백업 파일을 직접 복사하여 덮어쓰기합니다.

    # Windows CMD copy frontend\index.html.backup frontend\index.html # PowerShell / bash cp frontend/index.html.backup frontend/index.html

  3. 브라우저 캐시 강제 새로고침

    파일 복구 후 브라우저에 이전 버전이 캐시되어 있을 수 있습니다. Ctrl + Shift + R 로 강제 새로고침하여 최신 파일을 불러옵니다.

  4. 파일 위치 확인

    관련 파일 경로 안내:

    C:\Project\CDP-AI-Platform\ ├── frontend\ │ ├── index.html # 메인 파일 │ └── index.html.backup # 백업 파일 └── restore_frontend.bat # 복구 스크립트
